## 1. ВВЕДЕНИЕ

### 1.1 Актуальность работы

Современный ритм жизни городских жителей создаёт серьёзные препятствия для полноценного ухода за домашними питомцами. По данным исследований, более 60% владельцев собак в крупных городах испытывают трудности с регулярным выгулом своих питомцев из-за плотного рабочего графика, командировок или состояния здоровья.

Существующие решения на рынке PetTech преимущественно представлены агрегаторами услуг, которые связывают владельцев с человеческими исполнителями — выгульщиками. Однако такой подход имеет ряд существенных недостатков: зависимость от человеческого фактора, непостоянство качества услуг, проблемы с доверием и безопасностью.

Развитие технологий робототехники, компьютерного зрения и генеративного искусственного интеллекта открывает возможности для создания принципиально нового решения — автономной роботизированной платформы для выгула собак. Такая система способна обеспечить стабильное качество услуг, круглосуточную доступность, полную прозрачность процесса прогулки и повышенный уровень безопасности для питомца.

Проект «Шарик-шеринг» направлен на создание инновационного сервиса, объединяющего преимущества IoT-мониторинга и автономной робототехники с современными возможностями ИИ для генерации контента и обеспечения безопасности.

### 1.2 Цель проекта

Разработка концепции и прототипа роботизированной платформы для автономного выгула собак с интегрированной системой компьютерного зрения для распознавания угроз и генеративным ИИ для автоматического формирования отчётов о прогулках.

### 1.3 Задачи проекта

Для достижения поставленной цели необходимо выполнить следующие задачи:

1. Провести анализ рынка существующих решений в сфере PetTech и сервисов по выгулу собак
2. Сформулировать функциональные и нефункциональные требования к разрабатываемой системе
3. Выполнить обзор технологий и инструментов, необходимых для реализации проекта
4. Организовать командную работу с использованием современных методологий разработки
5. Разработать прототип модуля компьютерного зрения для распознавания объектов и угроз
6. Создать демонстрационные рекламные материалы с использованием генеративного ИИ
7. Оценить перспективность и возможность полноценной реализации проекта

## 2. ОБЗОР ПРЕДМЕТНОЙ ОБЛАСТИ

### 2.1 Анализ рынка существующих решений

Команда провела комплексный анализ рынка PetTech, разделив конкурентов на две основные категории: агрегаторы услуг (с человеческими исполнителями) и технологические платформы (IoT/трекинг). Данное разделение критически важно для позиционирования проекта «Шарик-шеринг», который должен занять уникальную нишу на стыке этих подходов.

#### 2.1.1 Агрегаторы услуг (Human-based)

**Wag! (США)**

Сервис по выгулу собак по модели «on-demand» (по запросу). Ключевые функции включают быстрый подбор исполнителя, страхование прогулок до $1 млн и функцию «Vet Chat» для консультаций с ветеринарами. Среди недостатков — высокая комиссия для исполнителей (до 40%) и существенная зависимость от человеческого фактора.

**Rover (США, Европа)**

Крупнейший международный маркетплейс услуг по уходу за домашними животными. Использует алгоритмы машинного обучения для мэтчинга владельцев и исполнителей на основе рейтингов. Отличается глубокой системой верификации личности и развитой системой отзывов. Работает в более чем 10 странах мира.

**Собака-Гуляка (Россия)**

Нишевой российский сервис с упором на стандартизацию качества услуг. Все выгульщики проходят обязательное обучение у профессиональных кинологов. Использование CRM-системы для контроля маршрутов и времени позволяет минимизировать риски и обеспечивать прозрачность для владельцев.

**Dogsy (Россия)**

Крупная российская площадка с широкой базой ситтеров по всей стране. Основной упор сделан на систему рейтингов и доступность услуг в различных регионах. Предоставляет услуги не только по выгулу, но и по передержке животных.

#### 2.1.2 Технологические платформы (Tech-based)

**Tractive (Австрия)**

Эталонное решение в сфере IoT-мониторинга домашних животных. Использует GPS-трекинг, систему геозон и ИИ-аналитику для отслеживания активности и фаз сна питомца. Устройство крепится к ошейнику и передаёт данные в реальном времени. Является технологическим ориентиром для модуля мониторинга здоровья в проекте «Шарик-шеринг».

#### 2.1.4 Консолидированный анализ функций

**Общие функции всех рассмотренных систем:**

- **Матчинг:** Интеллектуальный подбор исполнителя на основе профиля питомца
- **Отчётность:** Фото-, видео- и GPS-отчёты о маршрутах прогулок
- **Безопасность:** KYC-процедуры (проверка документов), страхование и эскроу-платежи
- **Бронирование:** Управление расписанием через платформу

**Уникальные особенности и технический стек:**

- **CRM-системы:** Используются для событийного управления заказами (Event-driven architecture), что позволяет диспетчерам видеть отклонения от маршрута в реальном времени
- **Machine Learning:** Применяется в рекомендательных системах для мэтчинга. Алгоритмы анализируют исторические данные прогулок для минимизации рисков
- **Чат-боты и уведомления:** Реализованы через Webhooks для мгновенной коммуникации «Владелец — Исполнитель»

### 2.2 Требования к разрабатываемой системе

На основе проведённого анализа рынка и выявленных недостатков существующих решений были сформулированы требования к системе «Шарик-шеринг».

#### 2.2.1 Функциональные требования

**Регистрация и управление пользователем:**

- Система должна обеспечивать регистрацию и авторизацию владельцев собак по номеру телефона, e-mail или через сторонние сервисы
- Система должна хранить и обрабатывать профиль пользователя, включая контактные данные, адреса и предпочтения по прогулкам
- Система должна позволять пользователю добавлять и редактировать профиль питомца

**Профиль питомца должен содержать:**

- Кличку
- Породу
- Возраст
- Вес
- Особенности поведения
- Уровень физической активности
- Ограничения по здоровью

**Система должна предоставлять возможность выбора параметров прогулки:**

- Длительность
- Время начала
- Интенсивность маршрута
- Индивидуальный сценарий прогулки

**Построение и управление маршрутом:**

- Система должна автоматически строить маршрут выгула на основе текущей геолокации пользователя и доступной городской инфраструктуры
- Система должна учитывать при построении маршрута: безопасные зоны для прогулки, плотность движения, наличие потенциально опасных участков, индивидуальные особенности питомца
- Система должна динамически перестраивать маршрут в режиме реального времени при изменении внешних условий
- Система должна сохранять историю маршрутов и предоставлять пользователю доступ к данным о предыдущих прогулках

**Распознавание угроз и обеспечение безопасности:**

- Роботизированная платформа должна распознавать окружающие объекты с использованием сенсоров и компьютерного зрения
- Система должна идентифицировать: других собак, людей, транспортные средства, препятствия на маршруте
- Система должна выявлять потенциально опасные ситуации, включая: приближение агрессивных собак, резкое изменение обстановки, движение объектов по траектории пересечения
- При обнаружении угрозы система должна: уведомлять владельца, снижать скорость движения, выполнять обход опасной зоны, при необходимости инициировать экстренную остановку
- Система должна обеспечивать возможность удалённого вмешательства со стороны владельца или оператора

**Генеративный ИИ и медиасопровождение прогулки:**

- Система должна вести видеозапись прогулки в автоматическом режиме
- Система должна обеспечивать онлайн-трансляцию прогулки в мобильное приложение владельца
- Система должна автоматически формировать краткий видеоотчёт по итогам прогулки с использованием генеративного ИИ
- Система должна генерировать текстовый отчёт, содержащий: длительность прогулки, пройденное расстояние, уровень активности питомца, количество остановок, зафиксированные события и предупреждения
- Система должна предоставлять пользователю базовую аналитику о состоянии и активности питомца

**Платежи и биллинг:**

- Система должна обеспечивать подключение банковской карты или другого электронного средства оплаты
- Система должна автоматически рассчитывать стоимость услуги в зависимости от выбранного тарифа и фактической длительности прогулки
- Система должна выполнять автоматическое списание оплаты после завершения оказания услуги
- Система должна формировать электронный чек и сохранять историю платежей
- Система должна поддерживать: разовые заказы, пакетные предложения, подписочную модель обслуживания

**Управление роботизированной платформой:**

- Система должна обеспечивать автономное управление роботом во время прогулки
- Робот должен самостоятельно следовать по маршруту, сохраняя безопасную скорость движения
- Робот должен контролировать уровень заряда аккумулятора и техническое состояние основных модулей
- При снижении уровня заряда ниже допустимого порога робот должен завершать прогулку и возвращаться на базу
- Система должна информировать пользователя о текущем статусе робота: в пути, прогулка начата, обнаружена угроза, прогулка завершена, возврат на базу, ошибка или неисправность

#### 2.2.2 Нефункциональные требования

**Надёжность и отказоустойчивость:**

- Система должна обеспечивать доступность цифровой платформы не ниже 99% времени
- Робот должен сохранять работоспособность в типовых городских погодных условиях
- Система должна обеспечивать безопасное завершение прогулки при потере связи с сервером или приложением
- Среднее время безотказной работы роботизированной платформы должно составлять не менее 1000 часов
- Критические функции безопасности должны иметь резервные сценарии исполнения

**Производительность и быстродействие:**

- Время первичного построения маршрута не должно превышать 5 секунд
- Время реакции системы распознавания угроз не должно превышать 1 секунду
- Задержка видеотрансляции в мобильное приложение не должна превышать 3 секунд
- Время генерации итогового видеоотчёта после прогулки не должно превышать 5 минут
- Интерфейс мобильного приложения должен обеспечивать плавную работу без заметных задержек

**Безопасность данных:**

- Персональные данные пользователей должны храниться в зашифрованном виде
- Передача данных между клиентским приложением, сервером и роботизированной платформой должна осуществляться по защищённым протоколам
- Система должна обеспечивать разграничение прав доступа для пользователей, операторов и администраторов
- Сервис должен соответствовать требованиям законодательства о защите персональных данных
- Доступ к платёжной информации должен быть защищён в соответствии с отраслевыми стандартами безопасности

**Автономность и энергопотребление:**

- Время автономной работы робота должно составлять не менее 2–3 часов непрерывной прогулки
- Система должна контролировать уровень заряда в реальном времени
- При достижении критического уровня заряда робот должен автоматически инициировать возврат на станцию
- Полная зарядка роботизированной платформы не должна превышать 2 часов

**Масштабируемость и сопровождаемость:**

- Архитектура системы должна поддерживать подключение новых модулей без полной переработки платформы
- Система должна обеспечивать возможность масштабирования на несколько районов города и на несколько городов
- Серверная часть должна поддерживать рост числа одновременных пользователей без критического снижения производительности
- Программное обеспечение должно поддерживать централизованное обновление компонентов

**Удобство использования:**

- Интерфейс мобильного приложения должен быть интуитивно понятным для пользователя без специальной подготовки
- Основные действия пользователя должны выполняться не более чем за 2–3 шага
- Система должна предоставлять push-уведомления о ключевых событиях прогулки
- Время освоения базового функционала пользователем не должно превышать 10 минут

## 3. ОБЗОР ИНСТРУМЕНТОВ И ТЕХНОЛОГИЙ

### 3.1 Общие технологии для реализации проекта

#### 3.1.1 Backend-разработка

**Python 3.11 + FastAPI**

Для обработки телеметрии от робота и статусов прогулок необходим быстрый асинхронный сервер. FastAPI идеально подходит для этой задачи — фреймворк обеспечивает высокую производительность и легко масштабируется. Версия Python 3.11 даёт значительный прирост в скорости работы по сравнению с предыдущими версиями благодаря оптимизациям интерпретатора.

**Pydantic**

В системе присутствует большое количество критичных данных: профили собак с информацией о здоровье, GPS-координаты, статусы заряда роботов. Библиотека Pydantic обеспечивает строгую валидацию этих данных на уровне типов, предотвращая ситуации, когда неверный тип переменной может привести к сбою сервера во время прогулки.

**PostgreSQL + PostGIS**

PostgreSQL — наиболее надёжная реляционная СУБД с открытым исходным кодом. Для проекта критически важно наличие расширения PostGIS, которое превращает базу данных в мощный геопространственный движок. Это позволяет эффективно решать задачи хранения маршрутов, поиска ближайших объектов и динамического перестроения путей.

#### 3.1.2 Frontend-разработка

**HTML5, CSS3, JavaScript (ES6)**

Стандартный стек веб-технологий для создания пользовательского интерфейса. Современный JavaScript позволяет создавать быстрые и отзывчивые интерфейсы. При разработке используются безопасные методы работы с DOM для предотвращения XSS-уязвимостей.

**TailwindCSS**

Утилитарный CSS-фреймворк, позволяющий быстро собирать адаптивный интерфейс с помощью utility-классов непосредственно в HTML-разметке. Значительно ускоряет процесс прототипирования и обеспечивает консистентность визуального оформления.

**OpenStreetMaps API**

Открытое решение для работы с картографическими данными и построения маршрутов. В отличие от коммерческих аналогов, OpenStreetMaps предоставляет бесплатный доступ к данным и позволяет накладывать собственные слои (например, опасные зоны или данные о плотности движения).

#### 3.1.3 Хостинг и деплой

**GitHub Pages / Vercel**

Для обеспечения доступности фронтенда не ниже 99% используются современные платформы статического хостинга. Vercel и GitHub Pages предоставляют глобальный CDN, автоматический деплой при обновлении кода и бесплатный SSL-сертификат.

### 3.2 Технологии для реализации ИИ-функциональности

#### 3.2.1 Компьютерное зрение

**YOLOv8 (You Only Look Once, версия 8)**

Для выполнения требования о времени реакции на угрозы менее 1 секунды выбрана модель семейства YOLO. Это индустриальный стандарт для распознавания объектов в реальном времени. YOLOv8 способна идентифицировать:

- Других собак (с определением породы и размера)
- Людей (с оценкой направления движения)
- Транспортные средства
- Препятствия на маршруте

Восьмая версия модели отличается оптимизированной архитектурой, что позволяет запускать её на встроенных вычислительных модулях робота без использования облачных сервисов.

**Ultralytics Python SDK**

Официальная библиотека для работы с моделями YOLO, обеспечивающая удобный API для загрузки моделей, выполнения inference и постобработки результатов.

#### 3.2.2 Генеративный ИИ

#### 5.3.2 Базовый код распознавания объектов

```python
from ultralytics import YOLO
import cv2
import numpy as np

# Загрузка предобученной модели YOLOv8
model = YOLO('yolov8n.pt')  # nano-версия для быстрого inference

# Классы объектов, которые нас интересуют
THREAT_CLASSES = {
    16: 'dog',      # Другие собаки
    0: 'person',    # Люди
    2: 'car',       # Автомобили
    7: 'truck',     # Грузовики
    5: 'bus',       # Автобусы
    1: 'bicycle',   # Велосипеды
    3: 'motorcycle' # Мотоциклы
}

def detect_threats(frame):
    """
    Распознавание объектов на кадре и определение угроз.

    Args:
        frame: numpy array, изображение с камеры робота

    Returns:
        threats: список обнаруженных угроз с координатами
    """
    # Выполнение inference
    results = model(frame, verbose=False)

    threats = []

    for result in results:
        boxes = result.boxes

        for box in boxes:
            cls_id = int(box.cls[0])
            conf = float(box.conf[0])

            # Фильтрация по интересующим классам
            if cls_id in THREAT_CLASSES and conf > 0.5:
                x1, y1, x2, y2 = box.xyxy[0].tolist()

                threat = {
                    'class': THREAT_CLASSES[cls_id],
                    'confidence': conf,
                    'bbox': [x1, y1, x2, y2],
                    'center': [(x1 + x2) / 2, (y1 + y2) / 2]
                }
                threats.append(threat)

    return threats


def calculate_threat_level(threats, frame_width, frame_height):
    """
    Расчёт уровня угрозы на основе обнаруженных объектов.

    Args:
        threats: список обнаруженных угроз
        frame_width: ширина кадра
        frame_height: высота кадра

    Returns:
        threat_level: строка ('low', 'medium', 'high', 'critical')
        actions: рекомендуемые действия
    """
    if not threats:
        return 'low', []

    actions = []
    max_level = 0

    for threat in threats:
        x1, y1, x2, y2 = threat['bbox']
        obj_width = x2 - x1
        obj_height = y2 - y1

        # Расчёт относительного размера объекта
        relative_size = (obj_width * obj_height) / (frame_width * frame_height)

        # Расчёт близости к центру кадра (траектории движения)
        center_x = threat['center'][0]
        distance_from_center = abs(center_x - frame_width / 2) / (frame_width / 2)

        # Определение уровня угрозы для объекта
        if threat['class'] == 'dog':
            # Другие собаки — потенциально высокая угроза
            if relative_size > 0.1:
                max_level = max(max_level, 3)
                actions.append('Обнаружена собака вблизи — снизить скорость')
            elif relative_size > 0.05:
                max_level = max(max_level, 2)
                actions.append('Обнаружена собака — наблюдение')

        elif threat['class'] in ['car', 'truck', 'bus', 'motorcycle']:
            # Транспорт — критическая угроза при близком расположении
            if relative_size > 0.15 and distance_from_center < 0.3:
                max_level = max(max_level, 4)
                actions.append('КРИТИЧНО: Транспорт на траектории — остановка')
            elif relative_size > 0.08:
                max_level = max(max_level, 3)
                actions.append('Транспорт вблизи — изменить маршрут')

        elif threat['class'] == 'person':
            # Люди — низкая угроза, но требуют внимания
            if relative_size > 0.2:
                max_level = max(max_level, 1)
                actions.append('Человек на пути — обойти')

    level_map = {0: 'low', 1: 'low', 2: 'medium', 3: 'high', 4: 'critical'}

    return level_map.get(max_level, 'low'), actions


def process_camera_feed():
    """Обработка видеопотока с камеры робота."""
    cap = cv2.VideoCapture(0)  # Или путь к видеофайлу

    while True:
        ret, frame = cap.read()
        if not ret:
            break

        # Распознавание угроз
        threats = detect_threats(frame)

        # Расчёт уровня угрозы
        h, w = frame.shape[:2]
        threat_level, actions = calculate_threat_level(threats, w, h)

        # Вывод информации
        print(f"Уровень угрозы: {threat_level}")
        for action in actions:
            print(f"  → {action}")

        # Визуализация (для отладки)
        for threat in threats:
            x1, y1, x2, y2 = [int(x) for x in threat['bbox']]
            if threat_level == 'low':
                color = (0, 255, 0)
            elif threat_level == 'medium':
                color = (0, 165, 255)
            else:
                color = (0, 0, 255)
            cv2.rectangle(frame, (x1, y1), (x2, y2), color, 2)
            cv2.putText(
                frame,
                f"{threat['class']} {threat['confidence']:.2f}",
                (x1, y1 - 10),
                cv2.FONT_HERSHEY_SIMPLEX,
                0.5,
                color,
                2
            )

        cv2.imshow('Threat Detection', frame)

        if cv2.waitKey(1) & 0xFF == ord('q'):
            break

    cap.release()
    cv2.destroyAllWindows()


if __name__ == "__main__":
    process_camera_feed()
```

#### 5.3.3 Модуль генерации отчётов с GigaChat

```python
from gigachat import GigaChat
from gigachat.models import Chat, Messages, MessagesRole


def generate_walk_report(walk_data: dict, gigachat_credentials: str) -> str:
    """
    Генерация текстового отчёта о прогулке с помощью GigaChat.

    Args:
        walk_data: словарь с данными о прогулке
        gigachat_credentials: авторизационные данные GigaChat API

    Returns:
        str: сгенерированный отчёт
    """
    prompt = f"""
    Сгенерируй дружелюбный отчёт о прогулке собаки для владельца.

    Данные прогулки:
    - Кличка собаки: {walk_data['dog_name']}
    - Длительность: {walk_data['duration_minutes']} минут
    - Пройденное расстояние: {walk_data['distance_km']} км
    - Средняя скорость: {walk_data['avg_speed']} км/ч
    - Количество остановок: {walk_data['stops_count']}
    - Обнаруженные события: {', '.join(walk_data['events'])}
    - Уровень активности: {walk_data['activity_level']}
    - Погода: {walk_data['weather']}

    Сделай отчёт тёплым и информативным, как будто пишешь заботливому хозяину.
    """

    with GigaChat(credentials=gigachat_credentials, verify_ssl_certs=False) as giga:
        response = giga.chat(
            Chat(
                messages=[
                    Messages(
                        role=MessagesRole.SYSTEM,
                        content="Ты помощник сервиса по выгулу собак. Пишешь отчёты для владельцев."
                    ),
                    Messages(
                        role=MessagesRole.USER,
                        content=prompt
                    )
                ],
                temperature=0.7,
                max_tokens=500
            )
        )

    return response.choices[0].message.content


# Пример использования
```

2. **Сформулированы требования к системе** — разработан полный набор функциональных и нефункциональных требований к платформе «Шарик-шеринг», включая требования к регистрации пользователей, построению маршрутов, распознаванию угроз, генерации отчётов, платежам и управлению роботом.

3. **Выполнен обзор технологий** — определён технологический стек для реализации проекта: Python/FastAPI для backend, PostgreSQL/PostGIS для хранения данных, YOLOv8 для компьютерного зрения, GigaChat API для генерации текстовых отчётов.

4. **Организована командная работа** — настроены процессы коммуникации (Telegram, Discord), управления задачами (GitHub Projects) и контроля версий (Git/GitHub). Проведены регулярные встречи и код-ревью.

5. **Разработан прототип модуля компьютерного зрения** — реализована система распознавания объектов и оценки угроз на базе YOLOv8, обеспечивающая время реакции менее 1 секунды (фактически ~45 мс).

6. **Созданы рекламные материалы** — с использованием генеративного ИИ (Midjourney, DALL-E 3, Runway) подготовлены визуальные материалы для презентации концепции проекта.

### 6.2 Выводы

Проведённый анализ и разработка прототипов подтверждают **техническую возможность** реализации проекта «Шарик-шеринг». Современные технологии компьютерного зрения (YOLOv8) обеспечивают достаточную скорость и точность для распознавания угроз в реальном времени. GigaChat API позволяет автоматизировать создание информативных отчётов о прогулках на русском языке с полным соблюдением требований российского законодательства.

**Перспективность проекта** обусловлена следующими факторами:

1. **Отсутствие прямых конкурентов** — на рынке нет решений, объединяющих автономную робототехнику и сервис по выгулу собак
2. **Растущий рынок PetTech** — увеличение числа домашних животных в городах и повышение требований к качеству ухода
3. **Развитие технологий** — снижение стоимости компонентов робототехники и повышение доступности ИИ-решений
4. **Социальные тренды** — увеличение занятости населения и потребность в автоматизации рутинных задач
